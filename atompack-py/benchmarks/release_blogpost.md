# Atompack: A Faster Storage Layer for Large Atomistic ML Datasets

_When atomistic datasets become training corpora instead of small scientific databases, the storage layer starts to matter a lot._

- Atompack is built for the point where shuffled reads, multiprocessing workers, and repeated dataset rebuilds start dominating the workflow.
- The maintained benchmark suite focuses on the workloads that usually hurt first in practice: random reads, multiprocessing scaling, write throughput, and memory behavior.
- The goal is not to replace ASE. It is to give atomistic ML pipelines a storage layer that feels much better once scale becomes part of the problem.

## 1. Why this problem exists

Large atomistic datasets are no longer unusual. Once a dataset moves from exploratory analysis into real training, the storage problem changes completely: the hot path is no longer a few manual lookups, but millions of shuffled reads, many workers, and memory behavior that stays under control over long runs.

That is where a gap often opens between scientific tooling and training infrastructure. ASE is excellent at what it is supposed to do. It is a strong interchange layer, it has the right abstractions for atomistic data, and it fits naturally into scientific Python workflows. But the backend that feels natural for research scripts is not automatically the backend you want when the dominant workload is `Dataset.__getitem__` under shuffle, across many epochs, with multiprocessing workers competing for I/O.

In practice, teams usually end up in one of two places. They either keep the convenience of ASE and accept substantial overhead in the training path, or they drop to custom raw LMDB pipelines to claw back throughput and storage efficiency. Neither outcome is especially satisfying: one leaves performance on the table, and the other often throws away too much ergonomics and interoperability.

Atompack is aimed directly at that gap.

## 2. What Atompack is

Atompack is a compressed atomistic dataset format and Python library built on a Rust storage engine. It stores atomistic structures together with the fields ML pipelines usually care about, including positions, species, forces, energies, charges, stress, cells, and custom properties. The design goal is simple: make the storage layer feel native to training workloads without making the Python surface painful to use.

It is not trying to replace the rest of the scientific Python stack. ASE remains the right abstraction for many parts of the workflow, and Atompack keeps a direct bridge to it through `from_ase(...)`, `to_ase(...)`, and batch conversion helpers. The pitch is narrower and more practical: when the bottleneck is how a large dataset is stored and read, especially under shuffled training access, Atompack should be a much better default than an ad hoc custom backend or an ASE-oriented storage path.

Under the hood, Atompack uses an append-only `.atp` format with per-record compression and a trailing index. The Python layer stays thin so that the expensive parts of storage, decompression, and indexing remain in Rust. For large read-only datasets, the open path can use a memory-mapped index to keep RAM overhead low while preserving fast random access. The result is a storage format that is much closer to what large atomistic ML training actually wants.

## 4. Baselines and why Atompack wins

The baseline set is intentionally mixed, because practitioners compare against different kinds of alternatives in real projects. `hdf5_soa` and `lmdb_soa` represent conventional array-oriented layouts: field-by-field storage, chunked along the molecule axis, and straightforward to reason about for fixed-shape materials data. `lmdb_packed` and `lmdb_pickle` represent the two common raw key-value directions people take when they want more control: a compact hand-packed binary record format on one side, and a flexible Python-native record payload on the other. `ase_sqlite` and `ase_lmdb` are ecosystem reference baselines, included not because they are supposed to be idealized high-performance storage engines, but because ASE-backed workflows are still what many users actually have today.

Those backends are not just weaker versions of the same idea. ASE-backed databases, in particular, give you something very valuable: a familiar mutable database surface, a natural scientific workflow, and a pleasant way to inspect, edit, and query structures during iteration. That is a real advantage. If the primary job of the storage layer is interactive dataset curation, small-scale updates, or scientific bookkeeping, that is a different optimization target than a read-heavy training corpus. Atompack offsets that loss in flexibility with much better iteration speed once the workload becomes repeated training access, but the tradeoff should be stated clearly: these tools are good at different things.

The same nuance applies to raw LMDB implementations. In principle, a very aggressive custom LMDB layout could be pushed further for one narrow metric. A team could hand-specialize the record format, add more fused batch paths, or optimize around a very specific consumer. But that comes with real complexity. The implementation surface gets larger, the format gets harder to reason about, interoperability usually gets worse, and there is still no guarantee that the resulting system will match Atompack across the full set of things that matter at once: write speed, random reads, multiprocessing behavior, storage size, memory footprint, and a usable Python-facing API.

Atompack should be explained as outperforming those alternatives for structural reasons, not as a benchmark accident. The `.atp` format is designed around append-only writes, per-record compression, and O(1) seeks through a trailing index. The read path is friendly to memory mapping, so large datasets can be reopened with low index overhead. Single-molecule reads use lazy views rather than eagerly rebuilding more Python objects than necessary, and the library exposes faster batch-oriented APIs when the consumer can use them. The punchline is straightforward: Atompack tries to remove Python-side overhead from the path that dataloaders hit constantly, while still keeping the API close enough to the surrounding ecosystem that it remains pleasant to use.

The practical adoption story should stay simple. Atompack is Apache-2.0 licensed, ships a Python package, interoperates with ASE, and already has a Hub-oriented path for publishing and reopening datasets through `atompack.hub`. That combination matters because storage formats only help if people can actually fit them into the workflows they already have. A faster backend is much more compelling when it does not force a full rewrite of the surrounding pipeline.

A short code example is enough:

```python
import atompack

db = atompack.Database.open("data.atp", mmap=True)
mol = db.get_molecule(0)
print(mol.energy)

atoms = mol.to_ase()
```

The conclusion should be direct and release-oriented. Atompack is a good fit when the storage layer itself has become a training bottleneck: large datasets, shuffled reads, multiple workers, and frequent conversions or exports. It is not a claim that every atomistic workflow should abandon ASE-backed tooling. It is a claim that once the workload is dominated by dataset access for ML, a storage format designed for that access pattern can make the whole system feel lighter, faster, and easier to scale.
